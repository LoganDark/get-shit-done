# Phase 2.1: VCS Abstraction Audit — Drop Git-Only Concepts (INSERTED) - Context

**Gathered:** 2026-05-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactor the cross-backend `VcsAdapter` type surface so it only exposes operations with a **direct jj equivalent**. Move git-only concepts to `vcs.gitOnly.*` (after `vcs.kind === 'git'` narrowing) or remove them entirely. Adopt **change-first identifier semantics** on the cross-backend surface (commit-hash on git, `change_id` on jj — never `commit_id` on jj). Mechanically refactor every caller to the new shape. **Still git-only — no jj backend code** (that's Phase 3).

This phase fixes what Phase 1's "forward-complete adapter" claim got wrong: not every adapter verb has a clean dual-backend story. Phase 2's bulk migration exposed surface that leaked git-specific concepts (stage/unstage, gitDir/gitCommonDir, StatusEntry.index, currentBranch terminology). Phase 2.1 cleans those up *before* Phase 3 lands jj against a flawed surface.

**In Phase 2.1:** type-surface reshape (`sdk/src/vcs/types.ts`, `sdk/src/vcs/expr.ts`), git backend update (`sdk/src/vcs/backends/git.ts`), all caller refactors (sdk + bin/lib), paired-test updates, baseline-parity dispatch updates, lint allowlist re-tightening.

**Not in Phase 2.1:** zero jj backend code (Phase 3); zero workspace/hook implementation (Phase 4); zero command/workflow markdown rewrites (Phase 5); ROADMAP success-criterion 5 rewording for Phase 4 is queued as a phase-transition task but not actioned in 2.1.

<decisions>
## Implementation Decisions

### Architectural Rule (the load-bearing principle)

- **D-01 (Strict surface):** The cross-backend `VcsAdapter` surface exposes **only** operations with a direct jj equivalent. Verbs requiring multiple-command translation internally are preferred over verbs with no jj analog. No git-only primitives leak onto the cross-backend type. Git-only verbs live under `vcs.gitOnly.*` after `vcs.kind === 'git'` narrowing, or are removed entirely.
- **D-02 (CommitInput collapse):** `CommitInput.files: string[]` is the single field for changed paths. Semantics shift from "stage these paths" to "**capture working-copy state for these paths**" — mix of adds/mods/dels. `undefined = all tracked changes`. The legacy `pathspec` field is removed.
- **D-03 (stage/unstage hard-remove):** `vcs.stage` and `vcs.unstage` are **removed from the adapter entirely** — not even moved to `gitOnly`. All 41 callers (34 stage + 7 unstage) refactor onto `vcs.commit({files})` with WC-state-capture semantics. A test that genuinely needs to probe git's index uses raw git inside an allowlisted test file, not the adapter.

### Commit & Deletion Semantics

- **D-04 (WC-state capture for deletes):** Deletions are recorded by deleting the file in the working copy and including its path in `commit.files`. The git backend synthesizes via `git add -A <paths>` or equivalent; the jj backend records the WC-state directly (`jj squash <paths> -B @ -k -m '<msg>'`). No `deletions` field on `CommitInput`; no backend-side inference via implicit status probe.
- **D-05 (`amend` stays cross-backend):** `CommitInput.amend: boolean` remains on the cross-backend surface. Git: `git commit --amend`. Jj: `jj squash --into @-` + `jj describe --no-edit` (Phase 3 details). 4 callers preserved.
- **D-06 (bug-2014 invariant becomes caller responsibility):** The `stagedOrUnstaged` short-circuit that Phase 2 plan 02-09 added to `cmdCommit` for #2014 (delete-missing-file regression) **stays in the caller, not the adapter**. With WC-state-capture semantics, `cmdCommit` pre-probes via `vcs.status()` and short-circuits to `nothing_to_commit` when no surviving entries; the adapter does not try to be smart about missing paths. Paired test `commit-files-deletion.test.cjs` re-asserts the invariant.

### Hooks & Verification

- **D-07 (Hooks internalized):** `vcs.hooks` is **removed from the public surface**. The `fireHook` helper (`sdk/src/vcs/hook-bridge.ts`) becomes a private module called from inside backend `commit` and `push` implementations only. No production caller exists today; tests in `git-backend.test.ts:210` and `adapter-contract.test.ts:74` are reframed (or removed).
- **D-08 (noVerify on the call surface):** `CommitInput.noVerify: boolean` and `PushOpts.noVerify: boolean` are the **only public knobs** for skipping hook firing. Cross-backend. Git: passes `--no-verify`. Jj: skips internal `fireHook` invocation post-squash / pre-push.
- **D-09 (Phase 4 HOOK-05 rewording REQUIRED):** Phase 4 ROADMAP success-criterion 5 currently reads "`vcs.hooks.fire('pre-commit', ctx)` is invoked after every `jj squash`..." — that wording assumes `vcs.hooks.fire` is a public verb. After D-07 lands, the criterion becomes **implementation language** (internal `fireHook` invocation) not type-surface language. ROADMAP edit task is queued for the Phase 2.1 → Phase 3 transition runner.
- **D-10 (Flag for Phase 4 verification — NOT decided in 2.1):** The ROADMAP and audit currently assume "colocated jj fires `.git/hooks/pre-commit` via colocation." This needs to be verified in Phase 4 because `jj squash` does not invoke `git commit-tree` — git's commit hooks may not fire automatically even in colocated mode. Phase 2.1 does not lock this; Phase 4 owns it.

### Identifier Semantics — Change-First on the Cross-Backend Surface

- **D-11 (Change-first project-wide):** The cross-backend identifier surface uses **commit hash on git, `change_id` on jj** — **never `commit_id` on jj**. Rationale: change_id is stable across rewrites (rebase/squash/amend), matches jj's idiomatic `jj log` default output, and aligns with the project's hard-fork purpose ("jj-first design with git as a backend"). Translators internally map change_id ↔ commit_id when shelling out to jj.
- **D-12 (Surfaces affected by D-11):** `LogEntry.hash` returns change_id on jj; `LogEntry.parents[]` returns change_id parents on jj; `vcs.refs.resolveShort(rev)` returns change_id prefix on jj; `expr.range(from, to)` resolves change_id range on jj; `vcs.refs.bookmarks.{create, move}(name, rev)` accepts change_id on jj.
- **D-13 (`expr.commit` renamed to `expr.rev`):** The runtime-id factory currently called `expr.commit(sha)` is renamed to `expr.rev(id)`. Validator is permissive enough to accept either git-hash alphabet (hex) or jj change_id alphabet; the translator dispatches based on backend. Mirrors `git rev-parse` and `jj -r REV` conventions. Maps cleanly to existing `RevisionExpr` type name.
- **D-14 (No `commit_id` escape hatch in 2.1):** Callers who genuinely need the underlying jj `commit_id` (e.g., for byte-level git-object interop in colocated mode) — defer to Phase 3. If a real need surfaces, Phase 3 adds a narrow `vcs.kind === 'jj'`-narrowed escape (`vcs.jjOnly.commitIdOf(change)` or similar). Phase 2.1 does not pre-emptively design it.

### Bookmark Surface

- **D-15 (`currentBookmarks(): string[]`):** Rename `vcs.refs.currentBranch()` → `vcs.refs.currentBookmarks()` and change return type from single string to **array**. Both backends can have 0..N bookmarks/branches pointing at the same revision; empty array means anonymous head (jj) or detached HEAD (git). 10 callers refactor.

### Status, Workspace Context, Ignore

- **D-16 (Drop `StatusEntry.index`):** With stage/unstage hard-removed (D-03), the `index` field on `StatusEntry` is meaningless on the cross-backend surface. `StatusEntry` becomes `{ path, worktree }`. Auditing callers happens during the refactor — sites that read `index` either become invalid (probably none, since there's no production use after Phase 2) or get reshaped onto `worktree`.
- **D-17 (`isIgnored` stays cross-backend with documented dual-semantic):** Bends D-01 deliberately. Both backends do read `.gitignore` semantically — git natively, jj on colocated *and* on jj-native (jj reads `.gitignore` when present even outside colocation). Documented dual-semantic is clean here: same input, same observable answer "is this path ignored". 6 callers unchanged. If jj-native ever surfaces a divergent `.jjignore`-only path, document and revisit; do not pre-emptively split.
- **D-18 (gitDir/gitCommonDir → gitOnly):** `WorkspaceContext` becomes `{ effectiveRoot, mode, isLinked }` cross-backend. Pure git paths `gitDir` and `gitCommonDir` move to `vcs.gitOnly.gitDir()` and `vcs.gitOnly.gitCommonDir()`. 5 + 6 callers narrow on `vcs.kind === 'git'`. `worktree-safety.cjs:122-123` (linked-worktree detection) is the primary consumer.

### init / configGet / configSet

- **D-19 (Stay gitOnly):** Despite jj having `jj init` / `jj git init` / `jj config get|set` equivalents, these stay under `gitOnly` because init flavor differs (`git init` vs `jj init` vs `jj git init --colocate` are three distinct ops) and config stores differ (`.git/config` vs `~/.jjconfig.toml` vs repo config). Bundling under `vcs.init()` / `vcs.config.*` would conflate semantics. Callers already know they're git-specific.

### Annotated Tags

- **D-20 (`createAnnotatedTag` stays gitOnly):** Confirmed. jj has lightweight tag refs only (colocated mode synthesizes git tags from bookmarks at push time). No change.

### Phase Mechanics

- **D-21 (Phase 2-style mechanical-only + long-lived branch):** Carry forward Phase 2's D-08 (mechanical-only invariant) and D-12/D-13 (long-lived `phase/2.1-audit` branch with broken-lint state during work, day-one allowlist re-tightening). Per-file commit granularity (D-05 from Phase 2), source+tests in same commit (D-06 from Phase 2), no surrounding-logic refactors. **Diverges from audit's suggestion to relax D-08** — discipline kept Phase 2 clean and rebase-friendly; the same logic applies. Verb-shape changes do need a single "shape commit" that touches the public types — that commit is naturally larger but stays mechanical-only in spirit (one type rename = one diff, no opportunistic edits).
- **D-22 (Allowlist re-tightening):** Phase 2 closed with the allowlist at its post-migration steady state. Phase 2.1 does not add new entries to the allowlist; if a refactor surfaces a new raw-git site that needs allowlisting, that's a bug in the audit (escalate, do not silently allowlist).

### Claude's Discretion

- **Plan numbering and exact ordering:** Planner picks plan boundaries within Phase 2.1. Suggested sequencing: (a) types.ts + expr.ts shape commit first; (b) git backend update to match new shape; (c) caller refactors in ascending LOC order (mirroring Phase 2 D-02); (d) `expr.commit → expr.rev` rename as a separate commit because it touches every range/exists/log site; (e) `currentBranch → currentBookmarks` rename as a separate commit.
- **Change-id ↔ commit-id translator placement (D-11/D-12):** Phase 2.1 reshapes the *type surface* and updates the git backend. The git backend doesn't need the translator (git only has one identifier). Phase 3 implements the jj-side translator inside `sdk/src/vcs/parse/jj-rev.ts` (or a new `sdk/src/vcs/parse/jj-id.ts`) — the plumbing site exists already from Phase 1.
- **Test fixture updates:** Tests that previously called `vcs.stage([files])` then `vcs.commit({message})` reshape onto `vcs.commit({files, message})`. Tests that probed `vcs.unstage()` reshape onto delete-file-in-WC + `vcs.commit({files: [path]})`. Mechanical pattern, planner allocates per-file.
- **#2014 caller-side check site:** `cmdCommit` in `bin/lib/commands.cjs` is the primary site (per Phase 2 plan 02-09). Planner may discover similar caller-side checks needed in other CLI flows; add as they surface.

</decisions>

## Specific Ideas

- **Branch name:** `phase/2.1-audit` (mirrors Phase 2's `phase/02-migration` naming convention, with the new `2.1-` prefix per filename convention).
- **First commit:** "Shape commit" — `types.ts` + `expr.ts` + `backends/git.ts` together. Lands the new public surface and the git backend's compliance with it. Every downstream caller is broken on the long-lived branch until that commit lands; from then on, each caller refactor is one commit, ascending LOC, mirroring Phase 2 D-02.
- **`expr.commit → expr.rev` rename as its own commit:** Touches every range/exists/log call site (9 sites in verify alone, plus progress/commands/graphify). Bundling into a single rename commit is cleaner than embedding the rename in each per-file commit. Mechanical-only, byte-equivalent semantics, instant to review.
- **`currentBranch → currentBookmarks` rename as its own commit:** 10 callers. Same reasoning as above. Note the type change (string → string[]) requires every caller to handle the array; planner inspects each site to ensure consumer logic survives (most likely take `[0]` or null-coalesce).
- **CommitInput reshape commit:** Bundles `pathspec` removal + `files`-semantics shift + `noVerify` field add. Touches `sdk/src/query/commit.ts` + the 4 amend callers + the 10 pathspec callers. Larger than a typical Phase 2 commit but mechanically justified.
- **`vcs.hooks` removal commit:** Drops `VcsHooks` namespace from `VcsAdapterCommon`, removes `hooks: VcsHooks` from `GitAdapter`'s return shape, updates `adapter-contract.test.ts:74` + `git-backend.test.ts:210` (delete or reframe tests). `fireHook` stays in `hook-bridge.ts` as an unexported helper.
- **`StatusEntry.index` drop commit:** Audits callers first (probably zero production reads after Phase 2's bug-2014 fix in `cmdCommit`), then drops the field. Quick commit.
- **`gitDir`/`gitCommonDir` move commit:** `WorkspaceContext` reshape + `gitOnly.gitDir()`/`gitCommonDir()` add + 11 caller refactors (worktree-safety.cjs:122-123 primary). Bundled because the type change forces every caller to update simultaneously.
