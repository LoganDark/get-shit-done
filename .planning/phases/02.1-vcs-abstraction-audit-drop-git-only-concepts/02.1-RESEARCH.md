# Phase 2.1: VCS Abstraction Audit — Drop Git-Only Concepts (INSERTED) - Research

**Researched:** 2026-05-11
**Domain:** Cross-backend VCS adapter type-surface reshape (TypeScript types + git backend + caller refactor; still git-only — no jj backend code)
**Confidence:** HIGH on call-site inventory and shape commits (grep-verified against current `main`). HIGH on decision-to-code mapping (each D-NN ties to a specific file). MEDIUM on baseline drift counts (~55 baselines exist; exact per-baseline drift requires shape-commit experimentation).

## Summary

Phase 2.1 reshapes `sdk/src/vcs/types.ts` + `sdk/src/vcs/expr.ts` + `sdk/src/vcs/backends/git.ts` to expose only operations with a direct jj equivalent, drops `vcs.stage` / `vcs.unstage` / `vcs.hooks` from the public surface, removes `pathspec` from `CommitInput`, renames `currentBranch → currentBookmarks` (string → string[]), renames `expr.commit → expr.rev` with change-first semantics, drops `StatusEntry.index`, moves `gitDir` / `gitCommonDir` to `vcs.gitOnly.*`, and mechanically refactors every caller. No new functionality lands — the existing git backend behavior is preserved while the type surface is jj-ready.

The phase is unusual in two ways: (1) it is *strictly negative* (removing concepts) rather than additive, and (2) the load-bearing change is a single "shape commit" that breaks every downstream caller's TypeScript at once — every subsequent commit is a caller fix. This mirrors Phase 2's pattern (long-lived branch with broken-lint state, per-file commits in ascending LOC order) but with the central type-shape change happening *first* instead of incrementally.

**Primary recommendation:** Land one shape commit (`types.ts` + `expr.ts` + `backends/git.ts`) that introduces the entire new surface simultaneously, then refactor callers in dedicated per-concern commits (one for `expr.commit → expr.rev`, one for `currentBranch → currentBookmarks`, one for `CommitInput` reshape, one for `vcs.hooks` removal, one for `gitDir/gitCommonDir → gitOnly`, one for `StatusEntry.index` drop). Per-file ascending-LOC commits inside each concern group. Use long-lived branch `phase/2.1-audit`, broken-lint accepted during work, lint clean before merge to `main`.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**D-01 (Strict surface):** Cross-backend `VcsAdapter` exposes ONLY operations with a direct jj equivalent. Verbs requiring multiple-command translation internally are preferred over verbs with no jj analog. No git-only primitives leak onto the cross-backend type. Git-only verbs live under `vcs.gitOnly.*` after `vcs.kind === 'git'` narrowing, or are removed entirely.

**D-02 (CommitInput collapse):** `CommitInput.files: string[]` is the single field for changed paths. Semantics shift from "stage these paths" to "**capture working-copy state for these paths**" — mix of adds/mods/dels. `undefined = all tracked changes`. The legacy `pathspec` field is **removed**.

**D-03 (stage/unstage hard-remove):** `vcs.stage` and `vcs.unstage` are **removed from the adapter entirely** — not even moved to `gitOnly`. All callers refactor onto `vcs.commit({files})` with WC-state-capture semantics. Tests that genuinely need to probe git's index use raw git inside an allowlisted test file, not the adapter.

**D-04 (WC-state capture for deletes):** Deletions are recorded by deleting the file in the working copy and including its path in `commit.files`. Git backend synthesizes via `git add -A <paths>` or equivalent; jj backend (Phase 3) uses `jj squash <paths> -B @ -k -m '<msg>'`. No `deletions` field on `CommitInput`; no backend-side inference via implicit status probe.

**D-05 (`amend` stays cross-backend):** `CommitInput.amend: boolean` remains on the cross-backend surface. Git: `git commit --amend`. Jj: `jj squash --into @-` + `jj describe --no-edit` (Phase 3 detail).

**D-06 (bug-2014 invariant becomes caller responsibility):** The `stagedOrUnstaged` short-circuit Phase 2 plan 02-09 added to `cmdCommit` for #2014 (delete-missing-file regression) **stays in the caller, not the adapter**. With WC-state-capture semantics, `cmdCommit` pre-probes via `vcs.status()` and short-circuits to `nothing_to_commit` when no surviving entries; the adapter does not try to be smart about missing paths. Paired test `commit-files-deletion.test.cjs` re-asserts the invariant.

**D-07 (Hooks internalized):** `vcs.hooks` is **removed from the public surface**. The `fireHook` helper (`sdk/src/vcs/hook-bridge.ts`) becomes a private module called from inside backend `commit` and `push` implementations only. No production caller exists today; tests in `git-backend.test.ts:210` and `adapter-contract.test.ts:74` are reframed (or removed).

**D-08 (noVerify on the call surface):** `CommitInput.noVerify: boolean` and `PushOpts.noVerify: boolean` are the **only public knobs** for skipping hook firing. Cross-backend. Git: passes `--no-verify`. Jj: skips internal `fireHook` invocation post-squash / pre-push.

**D-09 (Phase 4 HOOK-05 rewording REQUIRED):** Phase 4 ROADMAP success-criterion 5 wording becomes implementation language (internal `fireHook` invocation) not type-surface language. ROADMAP edit task is queued for the Phase 2.1 → Phase 3 transition runner.

**D-10 (Flag for Phase 4 verification — NOT decided in 2.1):** Whether colocated jj fires `.git/hooks/pre-commit` via colocation needs Phase 4 verification. Phase 2.1 does not lock this.

**D-11 (Change-first project-wide):** Cross-backend identifier surface uses **commit hash on git, `change_id` on jj** — **never `commit_id` on jj**. Translators internally map change_id ↔ commit_id when shelling out to jj.

**D-12 (Surfaces affected by D-11):** `LogEntry.hash` returns change_id on jj; `LogEntry.parents[]` returns change_id parents on jj; `vcs.refs.resolveShort(rev)` returns change_id prefix on jj; `expr.range(from, to)` resolves change_id range on jj; `vcs.refs.bookmarks.{create, move}(name, rev)` accepts change_id on jj.

**D-13 (`expr.commit` renamed to `expr.rev`):** The runtime-id factory currently called `expr.commit(sha)` is renamed to `expr.rev(id)`. Validator is permissive enough to accept either git-hash alphabet (hex) or jj change_id alphabet; the translator dispatches based on backend.

**D-14 (No `commit_id` escape hatch in 2.1):** Callers needing the underlying jj `commit_id` — defer to Phase 3. If a real need surfaces, Phase 3 adds a narrow `vcs.kind === 'jj'`-narrowed escape (`vcs.jjOnly.commitIdOf(change)` or similar).

**D-15 (`currentBookmarks(): string[]`):** Rename `vcs.refs.currentBranch()` → `vcs.refs.currentBookmarks()` and change return type from single string to **array**. Empty array means anonymous head (jj) or detached HEAD (git).

**D-16 (Drop `StatusEntry.index`):** With stage/unstage hard-removed (D-03), the `index` field on `StatusEntry` is meaningless on the cross-backend surface. `StatusEntry` becomes `{ path, worktree }`.

**D-17 (`isIgnored` stays cross-backend with documented dual-semantic):** Bends D-01 deliberately. Both backends do read `.gitignore` semantically. 6 callers unchanged.

**D-18 (gitDir/gitCommonDir → gitOnly):** `WorkspaceContext` becomes `{ effectiveRoot, mode, isLinked }` cross-backend. Pure git paths `gitDir` and `gitCommonDir` move to `vcs.gitOnly.gitDir()` and `vcs.gitOnly.gitCommonDir()`. `worktree-safety.cjs:97-99` (linked-worktree detection) is the primary consumer.

**D-19 (`init` / `configGet` / `configSet` stay gitOnly):** Init flavor and config stores differ enough that bundling under cross-backend `vcs.init()` / `vcs.config.*` would conflate semantics. Callers already know they're git-specific.

**D-20 (`createAnnotatedTag` stays gitOnly):** Confirmed. jj has lightweight tag refs only.

**D-21 (Phase 2-style mechanical-only + long-lived branch):** Carry forward Phase 2's D-08 (mechanical-only invariant) and D-12/D-13 (long-lived `phase/2.1-audit` branch with broken-lint state during work, day-one allowlist re-tightening). Per-file commit granularity (D-05 from Phase 2), source+tests in same commit (D-06 from Phase 2), no surrounding-logic refactors. Verb-shape changes need a single "shape commit" — naturally larger but mechanical-only in spirit.

**D-22 (Allowlist re-tightening):** Phase 2.1 does not add new entries to the allowlist. If a refactor surfaces a new raw-git site that needs allowlisting, that's a bug in the audit (escalate, do not silently allowlist).

### Pattern 1: Shape Commit + Caller Sweep
**What:** Single atomic commit lands every breaking type change at once; each subsequent commit fixes consumers for one concern.
**When to use:** Whenever a type-surface refactor would otherwise produce an unbuildable intermediate state. The alternative (introduce new shape additively, migrate callers, delete old shape) doubles the diff and gives Phase 2.1 no semantic benefit because the *whole point* is to remove old shape.
**Example:** Phase 1 D-04 + Phase 2 LEARNINGS show every prior type extension (commitInput.amend/noVerify/pathspec, workspace.context.gitDir, expr.commit factory) landed atomically with consumers. This is the same pattern, run backwards (removing instead of adding).

### Pattern 2: Concern-Grouped Cross-File Commits
**What:** A single commit handles ALL sites for ONE concern across multiple files, rather than ALL concerns in ONE file across multiple commits.
**When to use:** When the concerns are independent of each other but cross many files. Phase 2's pattern was the inverse (per-file, multi-concern) because each `execSync('git …')` swap was uniform — Phase 2.1's concerns are heterogeneous (rename / type change / field drop / verb drop), so concern-grouping reads better.
**Source:** CONTEXT.md specifics (`expr.commit → expr.rev as its own commit`, `currentBranch → currentBookmarks as its own commit`, `CommitInput reshape commit`, `vcs.hooks removal commit`).

### Pattern 3: Test Reshape Precedent (Phase 2 D-06)
**What:** Paired source-and-test commits stay atomic. A commit that changes `types.ts` plus `git.ts` is paired with its corresponding `adapter-contract.test.ts` / `git-backend.test.ts` updates in the same commit when feasible.
**When to use:** Whenever a surface change has a corresponding contract test. The shape commit (01) naturally bundles `adapter-contract.test.ts` and `git-backend.test.ts` updates because they sit next to `git.ts`. Per-file caller refactors bundle their `*.test.cjs` paired tests.
**Source:** Phase 2 D-06 (carried), 02-LEARNINGS.md Pattern: Per-File Commit + Paired-Test Atomic Shape.

### Anti-Patterns to Avoid

- **Don't move stage/unstage to gitOnly.** D-03 hard-removes them; moving to `gitOnly` defeats the purpose ("staging is git-only by definition; jj has no equivalent; we want callers to think in WC-state-capture terms, not staging terms").
- **Don't bundle deferred ROADMAP edits into a 2.1 plan.** Phase 4 HOOK-05 rewording (D-09) is queued for the next phase-transition runner, mirroring how Phase 2 plan 02-12 handled MIGR-04/UPSTREAM-01.
- **Don't introduce a `commit_id` escape hatch "just in case."** D-14 defers this to Phase 3 if real need surfaces. Pre-emptive design pollutes the surface with optionality before any caller demands it.
- **Don't relax mechanical-only invariant during the shape commit.** D-21 explicitly diverges from the audit's suggestion to relax. The shape commit will be large naturally (typedef rename + matching backend impl + matching contract tests); it stays mechanical-only in spirit (one rename = one diff line, no opportunistic edits).
- **Don't migrate `expr.commit → expr.rev` per-caller-file.** The CONTEXT.md specifics call for a single dedicated rename commit across all 47 invocations. Per-file would leak the rename across every other concern's commit.

## Common Pitfalls

### Pitfall 1: Stale dist-cjs after shape commit
**What goes wrong:** Shape commit lands `types.ts` + `git.ts` changes; but `sdk/dist-cjs/vcs/` is the artifact `bin/lib/*.cjs` requires at runtime. If `pretest`/build hooks don't run between the shape commit and the next test invocation, tests fail with mismatched runtime shapes.
**Why it happens:** Phase 2 LEARNINGS Surprise #8 (vitest spawn overhead) shows tests are run frequently in the per-file commit loop. Phase 1 D-02 routes unit tests through TS source (no rebuild needed) but integration tests through `dist-cjs/`.
**How to avoid:** Confirm `pnpm -F sdk build:cjs` runs in CI's pretest pipeline; spot-check that locally after the shape commit a `pnpm test:integration` rebuilds.
**Warning signs:** Test failures of the shape `TypeError: vcs.stage is not a function` or `currentBranch is not a function` in integration tests but not unit tests.

### Pitfall 2: Phase 2 LEARNINGS Surprise #1 redux — "complete" surfaces surface more gaps under execution
**What goes wrong:** Phase 2's "forward-complete" claim from Phase 1 surfaced 17 gaps + 3 blockers when callers were actually reshaped. Phase 2.1 may surface similar gaps when the new shape meets callers (e.g., a caller of `vcs.commit({pathspec:[...]})` may need a verb that doesn't exist after pathspec is removed).
**Why it happens:** Type-checking before-the-fact catches shape mismatches but not semantic gaps. A caller that conceptually needs "narrow commit scope without staging" (the old pathspec semantic) may have no clean translation to `files`-based WC-state-capture.
**How to avoid:** During the CommitInput reshape commit (04 in suggested ordering), trace each pathspec caller to confirm the semantic translates. CONTEXT.md D-02 says "Semantics shift from 'stage these paths' to 'capture working-copy state for these paths'" — the actual production caller `cmdCommit` (commands.cjs:382-384) already does upstream staging then commits; the pathspec there is *equivalent* to "commit the WC state of these paths after the explicit stage." The translation is mechanical, but verify each site.
**Warning signs:** A caller refactor commit can't be written mechanically — semantics require new helper logic in the caller. If this happens, surface as a research gap (planner inserts a 2.1-side gap-fill, NOT a new adapter verb per D-21 mechanical-only).

### Pitfall 3: Baseline drift after CommitInput reshape
**What goes wrong:** The 55 existing baselines under `tests/baselines/git-vcs/` include `commit-files-deletion`, `commit-ts-*-commit`, `commit-ts-*-add`, `commands-cjs-339-commit`, `commands-cjs-402-commit` snapshots that captured pre-reshape behavior. The reshape may not change git CLI invocations byte-for-byte, but if the backend's `commit()` body changes (e.g., `pathspec` branch removed), the dispatch in `baseline-parity.test.ts` must drop matching clauses.
**Why it happens:** `baseline-parity.test.ts` uses args-shape-keyed dispatch (Phase 2 LEARNINGS Pattern: Args-Shape Dispatch). Removing pathspec means several dispatch clauses no longer have anything to dispatch to. If left in place they become dead code; if removed but the JSON baselines stay, those baselines become unowned.
**How to avoid:** Step 08 in the suggested ordering (baseline-parity sweep) explicitly handles this: walk every baseline JSON, identify the args shape it represents, delete clauses for removed verbs (stage, unstage, hooks.fire), re-bless any baseline whose backend args changed.
**Warning signs:** `baseline-parity.test.ts` test count drops; orphan JSON files in `tests/baselines/git-vcs/`.

### Pitfall 4: `currentBookmarks(): string[]` empty-array vs null mismatch at consumer sites
**What goes wrong:** Phase 1 returned `string | null` ("null" = detached HEAD or anonymous head). Phase 2.1 changes to `string[]` (empty array = same condition). Consumers like `check-ship-ready.ts:49` (`current_branch = vcs.refs.currentBranch();`) currently take the value as-is and may compare to null. After the rename, they need `currentBookmarks()[0] ?? null` or `.length === 0` checks.
**Why it happens:** D-15 explicitly changes the return type. Each consumer must handle the array shape.
**How to avoid:** Step 03 in suggested ordering touches all 11 invocations. For each one, choose the consumer-correct translation:
- "show the current branch name in UI" → `bookmarks[0] ?? '<anonymous>'`
- "fail if detached" → `if (bookmarks.length === 0) throw …`
- "set HEAD-tracking branch" → loop over all bookmarks
**Warning signs:** TypeScript errors like `Type 'string[]' is not assignable to type 'string | null'`. The compiler is the primary safety net.

### Pitfall 5: `vcs.hooks` removal vs `fireHook` internal usage
**What goes wrong:** Step 07 removes `VcsHooks` from `VcsAdapterCommon`. The git backend at `backends/git.ts:467-469` currently exposes `hooks` as a frozen object that calls `fireHook(cwd, stage, ctx)`. After D-07, that frozen object disappears from the return shape; but `fireHook` must still be called from inside `commit()` and `push()` when `noVerify` is false.
**Why it happens:** D-07/D-08 internalize the helper but don't internalize the *invocation*. Currently `vcs.commit({noVerify: true})` skips hooks because the backend's `commit()` already appends `--no-verify` to git's argv. After D-07, the backend's `commit()` must ALSO call `fireHook(cwd, 'pre-commit', ctx)` *before* shelling out — because we are now in a world where `vcs.hooks.fire` is gone and the only path for pre-commit-hook firing is the internal invocation.
**Wait — does this apply to git?** Git's `git commit` natively fires `.git/hooks/pre-commit` already. The internal `fireHook` invocation is to fire `.githooks/<stage>` (the GSD-specific hook directory, NOT `.git/hooks/`). Re-check whether `.githooks/pre-commit` is currently being invoked from any production path before assuming this gap exists.
**How to avoid:** During step 07, audit whether production code today calls `vcs.hooks.fire(…)` at all (grep result: ZERO production callers — only adapter-contract and git-backend tests). If zero, the removal is purely cosmetic on the test side; no internal-invocation rewiring needed. CONTEXT.md D-07 specifies "becomes a private module called from inside backend `commit` and `push` implementations only" — verify Phase 4 (HOOK-01/HOOK-02/HOOK-03) is responsible for *adding* the internal call, not Phase 2.1.
**Warning signs:** If you find yourself adding `fireHook(…)` calls to `commit()` / `push()` during Phase 2.1, stop and confirm with CONTEXT.md — Phase 4 owns hook firing; 2.1 just changes the surface so Phase 4 can do it without a public verb.

### Concrete expr.rev rename (expr.ts excerpt)

```typescript
// BEFORE (expr.ts:99-104)
commit(sha: string): RevisionExpr {
  if (!SHA_HEX_RE.test(sha)) {
    throw new Error(`expr.commit: not a SHA-shaped string: '${sha}'`);
  }
  return brand(`commit:${sha}`);
},

// AFTER — D-13: rename + permissive validator
const SHA_OR_CHANGE_ID_RE = /^[0-9a-fA-F]{4,40}$|^[k-z]{4,40}$/;
// (jj change_id alphabet is k-z reversed-base32; git hex is 0-9a-f.
//  In Phase 2.1 only git is used at runtime, but the validator
//  pre-accepts the jj alphabet so Phase 3 doesn't need to relax it again.)

rev(id: string): RevisionExpr {
  if (!SHA_OR_CHANGE_ID_RE.test(id)) {
    throw new Error(`expr.rev: not a hex-SHA or change-id shaped string: '${id}'`);
  }
  return brand(`rev:${id}`);
},
```

The encoded prefix changes from `commit:` to `rev:` so `parseExpr` (and the per-backend translators `toGitRev` / `toJjRev`) must be updated to switch on `rev` instead of `commit`.
