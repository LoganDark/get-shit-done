## Area 3 — vcs.hooks.fire + noVerify placement

**Setup:** First-pass question framed "where does vcs.hooks.fire live (cross-backend / gitOnly / stub-on-jj)". User asked "what do we need hooks for?", which surfaced that the verb has **zero production callers today** and Phase 4 plans to fire hooks *internal* to vcs.commit/push. Question was reframed.

| Option | Description | Selected |
|--------|-------------|----------|
| Internalize: drop public vcs.hooks; add CommitInput.noVerify + PushOpts.noVerify | Move fireHook to internal helper. Backends call it from inside vcs.commit. Public knobs are noVerify flags only. | ✓ |
| Keep cross-backend public verb | vcs.hooks.fire stays public with dual-semantic per Phase 4 HOOK-05 as written. | |
| Move to gitOnly.hooks.fire | Strict D-01 application; loses non-colocated jj hook firing until v2. | |

**User's choice:** Internalize (Option 1).
**Notes:** Phase 4 ROADMAP success-criterion 5 needs rewording — currently reads as type-surface language, becomes implementation language. Queued for next phase transition. Also flagged: "colocated jj fires hooks via git" assumption in audit/ROADMAP needs Phase 4 verification — `jj squash` doesn't go through `git commit-tree`, so git hooks may not fire automatically.

| Option | Description | Selected |
|--------|-------------|----------|

**User's choice:** Three of four; the currentBranch and expr.commit items were reframed with a deeper terminology directive.

**Notes from user:**
> "currentBookmarks should be array; identifiers should always be changes on jj, not commits, so we should find a neutral term that works for both git and jj but refers to changes on jj"

This expanded into three sub-decisions:

### Area 4c.1 — currentBookmarks shape

| Option | Description | Selected |
|--------|-------------|----------|
| string[] (empty if anonymous/detached) | 0..N bookmarks/branches can point at the same revision. | ✓ |
| Different shape | Bookmark[] objects or tagged union. | |

### Area 4c.2 — Change-first identifier semantics

| Option | Description | Selected |
|--------|-------------|----------|
| change_id on jj everywhere | LogEntry.hash, resolveShort, expr factory id arg, range endpoints, bookmark target revs all use change_id on jj. | ✓ |
| Hybrid (specify per surface) | Some surfaces use change_id, others commit_id. | |
| Keep commit_id on jj across the surface | Reverts the push; Phase 1's documented stance. | |

### Area 4c.3 — Neutral factory name (replaces expr.commit)

| Option | Description | Selected |
|--------|-------------|----------|
