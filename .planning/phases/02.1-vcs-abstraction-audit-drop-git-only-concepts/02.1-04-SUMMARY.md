### Tracked, NOT auto-fixed — 3 deferred items (#3061 design tension)

**Three #3061 tests in commit.test.ts remain failing post-this-plan:**
- `commit named-files scope (#3061) > --files commits only the named paths when an unrelated change is pre-staged`
- `commit named-files scope (#3061) > .planning/ fallback commits only planning paths when an unrelated change is pre-staged`
- `commit named-files scope (#3061) > --amend with --files keeps the amend within the named-file scope`

**Root cause:** The #3061 invariant (pre-staged unrelated index entries must NOT leak into the commit) was previously enforced by the `-- <paths>` separator on `git commit` (the legacy partial-commit form). D-02 explicitly removes that separator: "the commit-scope-narrowing branch (formerly appending `-- <paths>` to `git commit`) is removed. Callers that need a narrow commit scope now pass `files` and rely on D-04 WC-state-capture semantics; the `-A` add stages exactly the requested paths and nothing more."

That last claim is incomplete: `git add -A -- <paths>` stages exactly the requested paths' WC state, but pre-existing index entries from outside that path set are NOT cleared. Without `-- <paths>` on `git commit`, those pre-existing index entries are swept into the commit.

**Why deferred:** Fixing requires a backend-side decision (Rule 4 territory):
- Option A: Reset the index for paths outside the requested set before `git add -A`. Surgical, narrow, preserves D-02's "WC-state for these paths" semantic. Cost: an extra `git reset HEAD -- <paths not in scope>` invocation, which is awkward to express (the set is the COMPLEMENT of the requested paths in the current index, not a fixed list).
- Option B: Re-introduce `-- <paths>` on `git commit` for the `files`-given branch. Restores #3061 exactly. Cost: contradicts D-02's stated removal of that branch, but D-02's rationale was about jj equivalence (`jj squash` doesn't need it), not about disabling #3061 protection in git.
- Option C: Accept the regression — under WC-state-capture, pre-staged unrelated entries DO leak in. Document as a known semantic shift; downgrade #3061 tests to `it.skip()` with an explanatory comment.

Plan-04 cannot make this decision unilaterally because Phase 2.1's success criteria require the #2014 invariant to survive (it does — see commit-files-deletion.test.cjs results) but say nothing about #3061. The #3061 protection was added in Phase 2 plan 02-09 as a hotfix; D-02's design intent in Phase 2.1 may not have considered the regression. Escalating to phase-level discussion is the right next step.

**Tracked:** Test file: `sdk/src/query/commit.test.ts` lines 297-396. Pre-plan-04 these same tests had infrastructure-level failures (vcs.stage not a function); post-plan-04 the infrastructure works and the test now exposes the genuine D-02 vs #3061 contract conflict.
