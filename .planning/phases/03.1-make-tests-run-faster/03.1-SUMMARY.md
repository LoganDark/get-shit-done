# Phase 03.1: make tests run faster

**The sdk/ vitest integration suite went from 16+ min (and intermittent hangs) to ~5 s per run and 7.4 s median total wall-clock. The win came from gating slow Claude-CLI tests behind an env var, not from any vitest config lever. Phase closed with 0 of 2 planned lever flips shipped.**

## What the phase actually delivered

| Surface | Before | After |
|---|---|---|
| `sdk/` vitest integration suite (default env) | ~16 min, intermittent hangs | ~5 s/run, deterministic |
| Baseline harness | did not exist | `sdk/scripts/profile-integration.mjs`, re-runnable |
| Pre-fix baseline data | did not exist | `.planning/intel/vitest-integration-baseline.md` with 3-run medians |
| Golden-parity tests (5 cases) | failing on `main` | passing |
| `lifecycle-e2e` / 2 phase-runner E2E tests | auto-ran whenever `claude` CLI present | opt-in via `GSD_ENABLE_E2E=1` |

## What the phase was originally chartered to deliver

Per ROADMAP: "Profile the slow `sdk/` vitest integration project and apply the largest wall-clock speedup achievable without breaking semantics" via lever flips on `sdk/vitest.config.ts` — specifically `pool: 'threads'` (L1) and `isolate: false` (L2).

Those levers were never flipped. The vitest config remains at its 22-LOC seed shape from Phase 03.

## Why the lever flips didn't ship

The pre-fix baseline data Plan 01 produced **showed the suite was already fast** once the slow tests were gated:

| File | Median wall-clock (3 runs) |
|---|---:|
| `golden.integration.test.ts` | 4109 ms |
| `read-only-parity.integration.test.ts` | 3259 ms |
| `phase-runner.integration.test.ts` (6 of 8 tests) | 13 ms |
| `sub-repos-root.integration.test.ts` | 9 ms |
| `e2e.integration.test.ts` (3 skipped) | 4 ms |
| `lifecycle-e2e.integration.test.ts` (1 skipped) | 0 ms |
| `init-e2e.integration.test.ts` (1 skipped) | 0 ms |
| **Total** | **7394 ms** |

At 7.4 s:
- Plan 02's "keep the flip if measured ratio > 1.0×" gate is statistical noise. Natural run-to-run variance is ±100–500 ms; a realistic L1 saving on 2 non-trivial files is in the same envelope. Decision boundary becomes coin-flip.
- Plan 03's `isolate: false` carries real correctness risk (cross-file module-scope state leakage) for a similarly marginal win — and the plan required a transitive-import audit before flipping, which would itself take longer than any savings.
- Plan 04 would have written up "no measurable change" either way.

User confirmation (2026-05-13, after running `pnpm vitest` directly in `sdk/`): "it seems to be of acceptable performance now."

Phase closed with per-plan SUMMARYs documenting reality (01 done; 02/03 skipped-by-design; 04 = this phase-close).

## The actual win

The user-visible 130× speedup (16 min → 5 s on the daily-friction `pnpm vitest` path) came from two pieces of work that weren't in the original phase plan:

1. **`GSD_ENABLE_E2E=1` gate on 3 slow tests.** The plan didn't anticipate that 3 tests would account for ~95% of integration wall-clock and that they'd be Claude-CLI-driven (and thus subject to API hangs). The convention `GSD_ENABLE_E2E` already existed for the `init-e2e` and `e2e` files; this phase aligned `lifecycle-e2e.integration.test.ts` and 2 cases in `phase-runner.integration.test.ts` to the same convention. Commit: `uxqknykpnlrvynmssypnzpotmrlzlvno`.

2. **Five SDK↔gsd-tools.cjs parity drift fixes.** Plan 01's D-05c gate (3-of-3 runs `success: true`) refused to record baseline while 5 deterministic golden-parity assertions failed. The 5 failures were independent SDK-port-lag bugs — upstream `bin/lib/*.cjs` PRs (#3178, #1971, #3242, #2836, #2009) that landed between Apr 29 and May 8 and never got mirrored into `sdk/src/query/`. Investigated via a `/gsd-debug` session (`golden-parity-failures`) and patched in one commit. Commit: `pxumponwopvsmlmvvlrusyqokxspskwv`. Debug record: `.planning/debug/resolved/golden-parity-failures.md`.

Without the parity fixes the harness couldn't have recorded a baseline. Without the test-gating the suite was unusably slow + occasionally hung. Both were unplanned but strictly upstream of Plan 01's commit gate.

## Issues encountered

1. **Run 2 of the harness hung at 14 minutes.** Vitest workers idle at 0% CPU; no further output. Caused by an LLM API call in `lifecycle-e2e` that didn't return (vitest's per-test timeout is 15 min, so it would eventually have fired — process killed at the harness's 30-min wall-clock cap). Resolved by gating those tests behind `GSD_ENABLE_E2E=1` (commit `uxqknykpnlrvynmssypnzpotmrlzlvno`).

2. **5 pre-existing parity failures (Phase 3 fallout).** Deterministic, not flaky. D-05c gate refused baseline. Resolved by porting the upstream CJS fixes (commit `pxumponwopvsmlmvvlrusyqokxspskwv`).

## User-visible changes

Running `pnpm vitest` (or the harness) on a checkout with `claude` CLI installed used to take 15+ min and could hang. Now ~5 s. Developers wanting the full E2E lane (CI nightly, manual regression) set `GSD_ENABLE_E2E=1` and pay the time cost intentionally.
