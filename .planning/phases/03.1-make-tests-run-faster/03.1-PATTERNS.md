**Analog 2 (secondary, for "drive a child process + parse + gate" shape):** `scripts/check-skip-count.cjs`

**Why:** It is the closest existing script that exercises the **gate** shape Plan 1's flakiness check (D-05c) wants: child-process driver + parse-output + count-comparison + exit-1-on-regression. It is `.cjs`, but the **control-flow shape** is what's load-bearing here, not the module system.

**Child-process driver shape** (`scripts/check-skip-count.cjs:72-88`):
```js
function countAllOnRef(ref) {
  let names;
  try {
    names = execSync('git ls-tree -r --name-only ' + ref, { cwd: REPO_ROOT, encoding: 'utf-8' }).split('\n').filter(Boolean);
  } catch {
    return null; // ref not available
  }
  const matching = names.filter((n) => /\.test\.(cjs|ts|mjs|js)$/.test(n) || /\.integration\.test\.(cjs|ts)$/.test(n));
  let total = 0;
  for (const f of matching) {
    let content;
    try {
      content = execSync('git show ' + ref + ':' + f, { cwd: REPO_ROOT, encoding: 'utf-8' });
    } catch {
      continue;
    }
    total += countSkips(content);
  }
  return total;
}
```
Apply equivalent: a `for (let i = 1; i <= 3; i++) { execSync('pnpm --filter @gsd-build/sdk exec vitest run --project integration --reporter=json --outputFile run-${i}.json'); }` loop. **But per RESEARCH "Baseline Statistical Protocol", the shell-loop primitive is cleaner than a Node-driven loop** — keep the loop in shell or in a small `.zsh` wrapper, and put the **aggregation** in the `.mjs` file. The driver and the aggregator can be two artifacts or one; planner's call.

**Counting + gate + diagnostic exit** (`scripts/check-skip-count.cjs:108-121`):
```js
if (current.total <= baseline) {
  console.log('ok check-skip-count: current=' + current.total + ' baseline(origin/main)=' + baseline);
  process.exit(0);
}

process.stderr.write('\nERROR check-skip-count: ' + (current.total - baseline) + ' new skip(s) since origin/main\n');
process.stderr.write('  current=' + current.total + '  baseline=' + baseline + '\n');
process.stderr.write('  Per-file skip counts (current branch):\n');
for (const f of current.perFile) {
  process.stderr.write('    ' + f.file + ' (' + f.count + ')\n');
}
process.exit(1);
```
**This is the exact shape of the D-05a test-count parity check** (RESEARCH Example 3): assert `numTotalTests === pre.numTotalTests` and `numPendingTests <= pre.numPendingTests`, diagnostic on regression, `process.exit(1)`.

**Language choice — `.mjs` vs `.ts`:** Per global CLAUDE.md "always use TypeScript for Node.js projects" / "tsx to execute TypeScript". The existing `sdk/scripts/` has both shapes:
- `sdk/scripts/gen-command-aliases.ts` (TypeScript, run by tsx — has a `tsx` consumer command somewhere)
- `sdk/scripts/check-command-aliases-fresh.mjs` (ESM, run by node directly, **runs after a build** so it can `await import('../dist/...')` from the built dist)
- `sdk/scripts/gen-profile-questionnaire-data.mjs` (ESM, **no build dep**, reads a `.cjs` file as plain text)

The baseline aggregator **does not need anything from the SDK build** (it only reads JSON files vitest writes). The natural fit is **`.mjs`** following `gen-profile-questionnaire-data.mjs` shape — no tsx, no build coupling. But TS+`tsx` (following `gen-command-aliases.ts`) is also acceptable. Planner picks; the global TS preference plus the project's mix of both styles makes either justifiable. Recommend `.mjs` for parsimony.

**Aggregator core shape** — see RESEARCH §"Example 2: Baseline aggregator (Node script outline)" for the median computation + markdown emission. That code is the spec for the inner loop of this file; the imports + assertion + exit shape come from `check-command-aliases-fresh.mjs`.

### `sdk/src/golden/inProcessGsdTools()` sibling (only if L4 selected) — CONDITIONAL CREATE

**Analog:** `sdk/src/golden/capture.ts` (the file it would sit next to)

**Why CONTEXT specifies this location:** L4 dispatch needs an in-process counterpart to `execGsdTools` (the spawn helper). Placing it as a sibling keeps the symmetry: `capture.ts` for subprocess (load-bearing for golden parity), and a sibling helper for in-process (used by the 3 non-golden files where L4 might land).

**Imports + helper-export shape** (`sdk/src/golden/capture.ts:1-13`):

```ts
 * Golden test helpers — run `gsd-tools.cjs` as a subprocess and capture JSON or raw stdout.
 *
 * Used by `golden.integration.test.ts` and `read-only-parity.integration.test.ts` to assert
 * SDK `createRegistry()` output matches the legacy CJS CLI.
 */

import { execFile } from 'node:child_process';
import { readFile } from 'node:fs/promises';
import { isAbsolute, join } from 'node:path';

import { resolveGsdToolsPath } from '../gsd-tools.js';
```

**Async helper-shape with reject-on-error** (`sdk/src/golden/capture.ts:17-49`):
```ts
function execGsdTools(
  projectDir: string,
  command: string,
  args: string[],
): Promise<{ stdout: string; stderr: string }> {
  const script = resolveGsdToolsPath(projectDir);
  const fullArgs = [script, command, ...args];
  return new Promise((resolve, reject) => {
    execFile(
      process.execPath,
      fullArgs,
      {
        cwd: projectDir,
        maxBuffer: MAX_BUFFER,
        timeout: CAPTURE_TIMEOUT_MS,
        env: { ...process.env },
      },
      (err, stdout, stderr) => {
        if (err) {
          ...reject(new Error(`gsd-tools failed (exit ${code}): ${stderrStr || ...}`));
          return;
        }
        resolve({ stdout: stdout?.toString() ?? '', stderr: stderr?.toString() ?? '' });
      },
    );
  });
}
```

**Exported wrapper that parses JSON** (`sdk/src/golden/capture.ts:76-83`):
```ts
export async function captureGsdToolsOutput(
  command: string,
  args: string[],
  projectDir: string,
): Promise<unknown> {
  const { stdout } = await execGsdTools(projectDir, command, args);
  return parseGsdToolsJson(stdout, projectDir);
}
```

**In-process counterpart shape** — would invoke `createRegistry().dispatch(command, args, projectDir)` (per RESEARCH §"Architectural Responsibility Map" and `sdk/src/query/sub-repos-root.integration.test.ts` which already calls this directly):

```ts
// Pattern shape (NOT verbatim from any file — synthesized from
//   - capture.ts:76-83 (export signature shape)
//   - sub-repos-root.integration.test.ts (already imports + calls createRegistry directly)
//   - read-only-parity.integration.test.ts:26-29 (registry.dispatch usage)
import { createRegistry } from '../query/index.js';

export async function inProcessGsdToolsOutput(
  command: string,
  args: string[],
  projectDir: string,
): Promise<unknown> {
  const registry = createRegistry();
  const result = await registry.dispatch(command, args, projectDir);
  return result.data;
}
```

**Critical caveat per RESEARCH A5 + L4 audit:** `GSDTools` already defaults `preferNativeQuery: true` (`sdk/src/gsd-tools.ts:68`). Most non-golden files **already** go in-process via that default. The L4 win is therefore bounded; this helper may end up unused. Planner decides per baseline.
