# Phase 03.1: make tests run faster - Context

**Gathered:** 2026-05-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Profile the `sdk/` vitest **integration** project (`sdk/src/**/*.integration.test.ts`, 7 files, `testTimeout: 120_000`), then apply the largest wall-clock speedup achievable without breaking semantics — both halves ship in this phase. Success is measured by ratio against a local baseline on the user's M-series machine.

**In Phase 03.1:**
- Per-file (and where useful, per-test) timing harness for the integration project, with a baseline report committed under `.planning/intel/`
- Mechanism selection driven by the baseline (planner picks from levers in D-06 based on what the profile reveals dominates)
- `sdk/vitest.config.ts` changes (e.g., `pool`, `isolate`, `poolOptions.*`) scoped to integration project where vitest supports per-project knobs
- In-process gsd-tools dispatch path for integration tests that are NOT explicitly subprocess-parity tests (i.e., NOT `golden.*` / `read-only-parity.*` — those exist to compare SDK vs subprocess and MUST keep spawning)
- Per-test fixture refactor of `sdk/src/vcs/__tests__/vcs-fixture.ts` style is permissible IF the profile shows describe-level fixture init dominates AND a less-invasive lever doesn't close the gap (rare for integration: integration tests don't use `vcsTest`, so the vcs-fixture pattern itself isn't on this phase's hot path — but a similar per-describe pattern may exist elsewhere)
- Final baseline + post-fix wall-clock + ratio recorded in the phase SUMMARY

**Not in Phase 03.1 (owned elsewhere):**
- `tests/*.test.cjs` cjs node:test suite speedups (455 files, separate runner, separate code path). The subprocess-spawn-cost work for that suite is its own future phase if pursued.
- `sdk/` vitest **unit** project (144 files) speedup. Out of scope — even though one informal baseline run of the unit project ran 15+ minutes with no output before being killed, the user has explicitly named integration as the daily friction. Unit speedup can spin off as a future phase using the harness this phase lands.
- CI runner tuning / matrix sharding / GitHub Actions cost work. Baseline is local-only (D-04). CI is informational.
- Backend matrix (`git` × `jj-colocated`) cost work — integration tests don't use `vcsTest`/`makeBackendFixture`, so the matrix axis doesn't multiply wall-clock here. Phase 4 will need its own answer for workspace test cost.
- Removing the `--ignore-working-copy` prohibition (locked by Phase 3 D-05) — speedups cannot rely on bypassing jj's auto-snapshot.

</domain>

<decisions>
## Implementation Decisions

### Scope

- **D-01 (Target = `sdk/` vitest integration project only):** Phase 03.1 optimizes `vitest run --project integration` (7 files: `init-e2e`, `lifecycle-e2e`, `phase-runner`, `e2e`, `golden/read-only-parity`, `golden/golden`, `query/sub-repos-root`). The cjs `tests/*.test.cjs` suite and the vitest unit project are explicitly out of scope. User locked this directly: "vitest is the pain point; it takes foreeever."
- **D-02 (Profile + fix in the same phase):** Plan 1 lands the profiling harness and commits a baseline report. Plans 2..N pick mechanisms based on what the baseline shows dominates, apply them, and verify ratio improvement. One phase, one ship — no follow-up phase to ship the actual speedup.
- **D-07 (Explicit non-goals):** No changes to `scripts/run-tests.cjs`, `tests/helpers.cjs`, the cjs node:test concurrency=4 setting, CI workflow files (`.github/workflows/test.yml`), or the backend matrix shape. If a planner-proposed mechanism would touch those, the planner must surface it and the scope decision returns here.

### Success Criterion

- **D-03 (Largest improvement without breaking semantics, ratio-vs-baseline):** No fixed minimum ratio. The phase ships the largest wall-clock reduction the planner can achieve while preserving semantics; the final ratio is recorded in the phase SUMMARY and reflected back to ROADMAP.md. Direction (smaller is better) is what matters.
- **D-05 ("Breaking semantics" = all of the following):**
  - **Test-count parity** — `vitest run --project integration --reporter=json` reports the same number of passed tests pre-change and post-change. Skip count must not increase. (Mirrors TEST-06's spirit on the vitest side; vitest has no built-in skip-count guard, so the verifier asserts this manually from the JSON reporter output.)
  - **Assertion fidelity** — no test bodies relaxed, no `.skip()` / `.todo()` / `.only()` added, no timeouts widened beyond the existing `testTimeout: 120_000` (which exists for a reason — long-running by design).
  - **No new flakiness** — final post-fix run is repeated 3× back-to-back; all three must pass. If any run fails or hangs, the most recent mechanism is reverted before the phase can close.
- **D-04 (Local M-series only):** Baseline and ratios are captured on the user's local machine (Darwin 25.5.0, 16 cores via `os.availableParallelism()`). CI numbers are not the gate. If a planner change would help CI but hurt local, local wins.

### Levers Available to the Planner

- **D-06 (Lever menu, planner selects based on profile evidence):**
  - **L1 — `pool: 'threads'`** in `sdk/vitest.config.ts` integration project. Vitest 3.x defaults to `'forks'` (process-per-file, ~200ms cold each); threads are ~30ms cold. For a 7-file suite where fork startup amortizes poorly, this is the cheapest lever. **No known semantic risk** for this suite — integration tests already use `child_process.execFile` for subprocess work, which is identical from a worker thread.
  - **L2 — `isolate: false`** (per-project). Reuses the worker context across files; vitest docs claim 2-5× on I/O-bound suites. **Risk:** module-level state leaks between files (top-level `let`, unstubbed mocks, registered signal handlers, module-cached singletons). The planner flips this only if the profile shows per-file module-init cost dominates, AND a test-isolation audit shows no offending top-level state in the 7 integration files.
  - **L3 — Per-test fixture refactor.** Only relevant if the integration suite shares a per-describe heavy fixture and the profile shows that's the bottleneck. Unlikely to apply: integration tests don't use the vcs-fixture pattern (no `vcsTest` / `makeBackendFixture` references in any integration file — verified by grep during discuss).
  - **L4 — In-process gsd-tools dispatch.** 5 of 7 integration files spawn `gsd-tools.cjs` via `execFile`. **`golden.integration.test.ts` and `read-only-parity.integration.test.ts` MUST keep spawning** — they exist to compare SDK vs subprocess output. The other 3 (`e2e`, `lifecycle-e2e`, `init-e2e`) and `phase-runner` (0 direct spawns but uses real GSDTools) can in principle invoke `createRegistry()` directly. Requires `gsd-tools.cjs` to be re-entrant (process.exit, env, cwd hazards) — non-trivial. Planner picks if profile points there.
  - **L5 — `test.concurrent` / `describe.concurrent`** within a file. Only if per-test fixtures land first AND profile shows within-file serialization is the bottleneck. Low likelihood for a 7-file suite; listed for completeness.
- **D-09 (Lever-vs-evidence rule):** Each lever flip must be tied to a specific bottleneck the baseline shows. The planner records, for each lever applied: which baseline observation justifies it, the predicted improvement, and the measured post-flip improvement. Flips that don't deliver are reverted before the next is applied. **No speculative knob-bumping** — even reversible knobs need evidence.

### Baseline Measurement Protocol

- **D-08 (Baseline = median of 3 runs, local, recorded to `.planning/intel/vitest-integration-baseline.md`):** The harness plan runs `vitest run --project integration --reporter=verbose` (or equivalent with per-file timings) three times, computes the median total wall-clock and per-file wall-clock, and commits the report. Post-fix verification reuses the same protocol so the ratio is apples-to-apples. The "3× back-to-back, all pass" flakiness gate from D-05 reuses these runs.

### Claude's Discretion

- **Profiling reporter shape** — planner picks: vitest built-in `--reporter=verbose`, custom `Reporter` class exported from the SDK, or a one-off `time pnpm …` wrapper script. Whatever yields per-file numbers that survive being committed to `.planning/intel/`.
- **Lever ordering** — D-06 lists 5 levers; D-09 demands evidence-tied flips but doesn't dictate which lever to try first. Planner picks the order based on what the baseline highlights.
- **Per-project `vitest.config.ts` shape** — the config currently has zero perf knobs. Planner picks whether to inline `poolOptions` into the project array entry or hoist shared knobs to the top-level `test:` block. Either is fine as long as the unit project remains untouched (D-01).
- **Recording final ratio location** — phase SUMMARY is mandatory; whether to also append a one-liner to ROADMAP.md or to `.planning/intel/` is planner's call.
- **Whether to expose vitest-perf knobs via env** — e.g., `VITEST_POOL=threads` as an escape hatch. Planner picks; not required.

</decisions>

## Existing Code Insights

### Reusable Assets
- **`sdk/src/gsd-tools.ts` `GSDTools` class + `resolveGsdToolsPath()`** — already exposed; L4 in-process dispatch can build on this rather than touching `gsd-tools.cjs` directly. The cjs entry already lives in `get-shit-done/bin/gsd-tools.cjs`.
- **`sdk/src/golden/capture.ts` `execGsdTools()`** — single point of spawn for golden tests; if L4 is applied selectively, this helper stays for golden but a sibling `inProcessGsdTools()` could land alongside.
- **`sdk/src/query/index.ts` `createRegistry()`** — the SDK's in-process query API; the natural target for L4.

### Established Patterns
- **Per-project vitest config in the projects array** (`sdk/vitest.config.ts`) — vitest 3.x supports per-project `pool`/`isolate`/`poolOptions`. Knob changes can be scoped to integration without affecting unit.
- **`pretest: pnpm run build:sdk`** — every test invocation rebuilds the SDK (tsc + tsc CJS). Not in scope for this phase but worth noting: a meaningful chunk of `time pnpm test` is fixed-cost rebuild. Bypassing it requires `--filter` discipline or a build cache (out of scope per D-07).
- **`testTimeout: 120_000` on integration project** — integration tests are designed to be long; speedups that reduce timeout headroom risk masking real bugs. The 120s ceiling stays.

### Integration Points
- `sdk/vitest.config.ts` is the single mutation point for L1, L2, L4-config-side knobs.
- `gsd-tools.cjs` re-entrancy (for L4) is a cross-cutting refactor — touches the cjs CLI's argv/env/exit handling. The planner may decide L4's cost outweighs payoff if profile shows other levers are sufficient.

</code_context>

## Specific Ideas

- **"Largest improvement without breaking semantics"** is the success criterion shape the user chose explicitly over `≥2×`, `≥3×`, and `≥1.5×` fixed ratios. The post-fix ratio number is the *output* of the phase, not its *target*.
- **Baseline credibility note:** during discuss-phase, an informal `time pnpm test -- --reporter=basic --project unit` invocation ran for 15+ minutes producing zero stdout before being killed. This is suggestive evidence that the underlying perf problem is real, but it's *unit*, not integration, and was killed mid-run; it does NOT establish an integration baseline. The harness plan must measure integration cleanly from scratch.
- **`--reporter=verbose` is the conventional vitest per-file timing surface.** Planner is free to pick a richer reporter if needed but the baseline file must contain per-file numbers, not just totals.
