# Phase 03.1: make tests run faster - Research

**Researched:** 2026-05-12
**Domain:** Vitest 3.x integration suite performance ‚Äî profiling harness + lever selection
**Confidence:** HIGH

## Summary

Phase 03.1 profiles 7 vitest integration files (`sdk/src/**/*.integration.test.ts`, ~1,851 LOC total) and applies the largest semantics-preserving speedup. Vitest 3.x's default pool is **`'forks'`** [VERIFIED: v3.vitest.dev/config/pool] ‚Äî fork startup amortizes poorly across only 7 files. The cheapest evidence-tied lever (L1 `pool: 'threads'`) is unblocked: zero `process.chdir` calls anywhere in `sdk/src/` or `get-shit-done/bin/` (verified by grep). The next lever (L2 `isolate: false`) is also low-risk: top-level mutable state in the 7 files is limited to import-time `cliAvailable` booleans that are written once inside try/catch (not re-written across tests), plus pure constants. The two golden files (`golden.integration.test.ts`, `read-only-parity.integration.test.ts`) are subprocess-parity tests by construction ‚Äî L4 (in-process dispatch) is permanently ineligible for them and must be flagged as such to the planner.

For the profiling harness, vitest 3.x's `--reporter=verbose` emits hierarchical per-test timings; `--reporter=json` emits a Jest-compatible schema with `testResults[].assertionResults[].duration` and per-file `startTime`/`endTime` [VERIFIED: v3.vitest.dev/guide/reporters]. The median-of-3 baseline protocol from D-08 is cheapest as a shell loop driving `--reporter=json` into a temp directory, then a small Node script aggregating per-file medians into the `.planning/intel/vitest-integration-baseline.md` markdown table.

**Primary recommendation:** Use `--reporter=json` (machine-parseable per-file/per-test timings) for the baseline harness; emit `--reporter=verbose` as a human-readable companion. Apply L1 (`pool: 'threads'`) first ‚Äî it is verified semantically safe for this suite and produces the largest cheap win. Defer L2 (`isolate: false`) until the L1-applied baseline shows per-file module-init still dominates. L4 stays as a high-cost option the planner reaches for only if the L1+L2 profile points there; the two golden files are permanently L4-ineligible.

### Pattern 1: Per-project pool override in vitest 3.x

**What:** Each entry in `test.projects[]` can carry its own `pool`, `isolate`, `poolOptions`, and `testTimeout`. Only `coverage`, `reporters`, and `resolveSnapshotPath` are documented as root-only [CITED: v3.vitest.dev/guide/projects].

**When to use:** Phase 03.1's whole point ‚Äî keep the unit project (144 files) on default forks+isolate while flipping integration (7 files) onto threads or isolate=false.

**Example (shape inferred from vitest 3.x docs; planner verifies on first build):**

```ts
// Source pattern: vitest 3.x projects + per-project pool/isolate per docs
// https://v3.vitest.dev/config/pool ‚Äî default pool: 'forks'
// https://vitest.dev/config/isolate ‚Äî isolate per-project supported
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    projects: [
      {
        test: {
          name: 'unit',
          include: ['src/**/*.test.ts'],
          exclude: ['src/**/*.integration.test.ts'],
          // unit project: leave at defaults (forks, isolate: true)
        },
      },
      {
        test: {
          name: 'integration',
          include: ['src/**/*.integration.test.ts'],
          testTimeout: 120_000,
          // L1: pool: 'threads',
          // L2: isolate: false,  // or per-pool: poolOptions.threads.isolate: false
          // poolOptions: { threads: { singleThread: false, isolate: true } },
        },
      },
    ],
  },
});
```

### Pattern 2: Baseline harness (median-of-3 with `--reporter=json`)

**What:** Shell-driven loop, JSON reporter output to temp files, Node aggregator computes median per file.

**When to use:** Plan 1's baseline + every post-flip verification (D-08).

**Example:**

```bash
# Source: harness shape derived from vitest 3.x --reporter=json schema
# https://v3.vitest.dev/guide/reporters ‚Äî JSON schema fields documented
set -euo pipefail
out=.planning/intel/_baseline-runs
mkdir -p "$out"
for i in 1 2 3; do
  pnpm --filter @gsd-build/sdk exec vitest run \
    --project integration \
    --reporter=json --outputFile "$out/run-$i.json" \
    --reporter=verbose 2>&1 | tee "$out/run-$i.verbose.txt"
done
node scripts/vitest-baseline-aggregate.mjs "$out" > .planning/intel/vitest-integration-baseline.md
```

The JSON reporter emits `testResults` array with per-file `startTime`/`endTime` and `assertionResults[].duration` [VERIFIED: v3.vitest.dev/guide/reporters]. Per-file wall-clock = `endTime - startTime`.

### Anti-Patterns to Avoid

- **Knob-bumping without baseline evidence.** D-09 forbids this. Every flip must point at a specific number in the baseline.
- **Treating `time pnpm test` as the baseline.** That measures wall-clock of `pnpm test` which includes `pretest: pnpm run build:sdk` (root `package.json:pretest`) ‚Äî confounds rebuild cost with test cost. Use `vitest run --project integration` directly after a single warm build.
- **Touching `golden.integration.test.ts` or `read-only-parity.integration.test.ts` spawn paths.** These tests exist to prove SDK in-process output equals CJS subprocess output [VERIFIED: `sdk/src/golden/golden.integration.test.ts:113` `expect(sdkResult.data).toEqual(gsdOutput)` where `gsdOutput` comes from `captureGsdToolsOutput(...)` which spawns `gsd-tools.cjs`]. Removing the spawn defeats the assertion's purpose.
- **Adding `.skip()` to dodge slow tests.** D-05 forbids skip-count increases. The JSON reporter's `numPendingTests` is the gate.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-file timing capture | Custom timer + reporter scaffolding | Built-in `--reporter=json` | Already emits per-file + per-test timings; Jest-compatible schema; one Node script to aggregate |
| Median computation across runs | Statistical-analysis library | 10-line Node script over the 3 JSON files | Median of 3 numbers needs `Math.median` (sort, pick middle) ‚Äî no dependency |
| Flakiness re-runs | Custom retry harness | Vitest built-in `--retry <n>` flag (default 0) [VERIFIED: v3.vitest.dev/guide/cli] | But: D-05 wants 3√ó back-to-back **independent** all-pass runs, not in-process retry ‚Äî keep this as a shell loop |
| Subprocess parity rewrite | Refactor `golden.integration.test.ts` to in-process | (don't) | The whole point of the file is subprocess vs SDK parity |
| Worker reuse for threads | Roll your own worker pool | `poolOptions.threads.{singleThread, maxThreads, minThreads, isolate}` [VERIFIED: v3.vitest.dev/config/pool] | Vitest exposes the knobs directly |

**Key insight:** The phase's surface is exactly two committable artifacts ‚Äî `sdk/vitest.config.ts` knob flips and `.planning/intel/vitest-integration-baseline.md` numbers. Everything else is shell + a tiny aggregator. Resist the urge to build a "timing framework."

### Gotcha: `--reporter=json` under `--project integration`

The JSON reporter is documented as root-level only when scoped at the workspace level (the `reporters` config option is one of the three documented root-only options [CITED: v3.vitest.dev/guide/projects: "coverage, reporters, resolveSnapshotPath are root-only"]). However, the **CLI flag** `--reporter=json` overrides this and applies to whatever project is being run. The planner should verify in a smoke test that `vitest run --project integration --reporter=json --outputFile x.json` actually produces output covering only the 7 integration files (no unit project contamination). This is the kind of thing that breaks silently.

Per-file wall-clock = `endTime - startTime`. Per-test = `assertionResults[].duration`. The fields the planner needs are stable across patches.

## L1 (`pool: 'threads'`) ‚Äî Audit

### Vitest 3.x defaults

- **Default pool:** `'forks'` [VERIFIED: v3.vitest.dev/config/pool ‚Äî "Default: 'forks'"].
- Valid values: `'threads' | 'forks' | 'vmThreads' | 'vmForks'` [VERIFIED: ibid].
- **Threads pool constraint** (CRITICAL): *"When using threads you are unable to use process related APIs such as `process.chdir()`."* [VERIFIED: v3.vitest.dev/config/pool].
- Forks pool: "Similar as threads pool but uses `child_process` instead of `worker_threads`. Process related APIs such as `process.chdir()` are available in forks pool." [VERIFIED: ibid].
- `poolOptions.threads`: `singleThread (false)`, `isolate (true)`, `maxThreads`, `minThreads` [VERIFIED: ibid].

### Per-project pool override syntax

Vitest 3.x supports per-project `pool` in the projects array. [CITED: per-project `isolate` is explicitly documented at v3.vitest.dev/config/isolate; per-project `pool` is supported because pool is not listed in the root-only set (`coverage`, `reporters`, `resolveSnapshotPath`) per v3.vitest.dev/guide/projects.] [ASSUMED]: that the `projects: [{ test: { pool: 'threads' } }]` shape works exactly like the top-level ‚Äî verified by smoke-testing in Plan 1 before committing.

### Semantic-risk audit for L1 against the 7 integration files

| Hazard | Source of concern | Audit result |
|--------|-------------------|--------------|
| `process.chdir()` calls | Threads pool forbids | **CLEAR** ‚Äî `grep -rE 'process\.chdir' sdk/src/ get-shit-done/bin/` returns 0 hits |
| Native modules requiring worker thread incompat | rare modules (e.g., better-sqlite3) | **CLEAR** ‚Äî sdk deps are `@anthropic-ai/claude-agent-sdk`, `ws`, `@types/node`, `@types/ws`, `typescript`, `vitest`. None are native. |
| Subprocess spawning | Threads pool fine with `child_process.execFile`; CONTEXT D-06 L1 already notes this | **CLEAR** ‚Äî `execFile` works the same from a worker thread |
| Top-level signal handlers | Worker threads forbid `process.on('SIGINT', ‚Ä¶)` | **CLEAR** ‚Äî no signal handler registration in the 7 files or transitive imports (grep verified) |
| `process.env` writes at module scope | Would race in threads pool with shared isolate=false | None at module scope in the 7 files; setup uses `vcs.gitOnly.configSet(...)` which doesn't mutate process.env |

**Cold-start cost data:** Tinypool benchmark via vitest-dev discussion #4914 ‚Äî *worker threads 10,371 ms vs child_process 12,100 ms across 2000 rounds with 8 threads* [CITED: github.com/vitest-dev/vitest/discussions/4914]. That's a ~14% steady-state delta, but for a 7-file suite where cold-start dominates over steady-state, the relative win is larger ‚Äî see "Prior Art" below.

**Conclusion:** L1 is unblocked on this suite. The planner can apply it without an isolation audit, because L1 leaves `isolate: true` (the default).

## L2 (`isolate: false`) ‚Äî Semantic Risk Audit

### What `isolate: false` changes

Default `isolate: true`: vitest creates a fresh module-graph per test file. [VERIFIED: v3.vitest.dev/config/isolate].
`isolate: false`: worker context (module cache, top-level state, `vi.mock` registrations) is reused across files in the same worker.

[CITED: v3.vitest.dev/config/isolate: "Disabling this option might improve performance if your code doesn't rely on side effects (which is usually true for projects with `node` environment)."]
[CITED: ibid: per-project disable supported via `projects` array.]

### `isolatedModules: true` + `isolate: false` interaction

`sdk/tsconfig.json:18` sets `"isolatedModules": true`. This is a **build-time** TS compiler flag that enforces each file is independently compileable (no `const enum`, no cross-file type-only `import` cycles). It has **no runtime semantics**. `isolate: false` is a **runtime** vitest flag that controls module-cache reuse in the worker. The two are orthogonal. [VERIFIED: TypeScript docs on isolatedModules; vitest docs on isolate.]

## L4 (In-process gsd-tools dispatch) ‚Äî Re-entrancy Audit

### Subprocess parity guard (CRITICAL)

`sdk/src/golden/golden.integration.test.ts` and `sdk/src/golden/read-only-parity.integration.test.ts` exist explicitly to compare SDK in-process output (`createRegistry().dispatch(...)`) with subprocess CJS output (`captureGsdToolsOutput(...)` which spawns `gsd-tools.cjs`).

Example load-bearing assertion: `sdk/src/golden/read-only-parity.integration.test.ts:26` ‚Äî
```ts

const registry = createRegistry();
const sdkResult = await registry.dispatch(row.canonical, row.sdkArgs, REPO_ROOT);

```

Removing the spawn defeats the parity claim. **These two files are permanently L4-ineligible.** Tag them as such in the plan.

| `sdk/src/init-e2e.integration.test.ts` | 0 direct; uses `GSDTools` (which already defaults to in-process registry per `sdk/src/gsd-tools.ts:67-69`: `preferNativeQuery: true`) | Partial ‚Äî already in-process when CLI available. L4 might already be implicitly active. |

| `sdk/src/query/sub-repos-root.integration.test.ts` | 0 ‚Äî calls `createRegistry()` directly already | **Already in-process** |

**Important nuance discovered:** `GSDTools` constructor [VERIFIED: `sdk/src/gsd-tools.ts:67-69`] defaults `preferNativeQuery: true`. So the 4 files that use `GSDTools` (init-e2e, lifecycle-e2e, phase-runner, and e2e indirectly through `GSD`) **already route through `registry.dispatch()`** for native commands, *not* through `gsd-tools.cjs` subprocess. The remaining subprocess spawns are inside `captureGsdToolsOutput` (golden files only) and `which claude` checks (cliAvailable at module load).

**Implication for the planner:** L4 may produce no win at all on the non-golden files because they don't spawn anymore. The cost-to-fix `gsd-tools.cjs` re-entrancy is **not** worth incurring for marginal gain. Confirm via baseline before committing to L4.

### `gsd-tools.cjs` re-entrancy hazards (for completeness, in case L4 is reached)

If L4 *is* selected, `gsd-tools.cjs` would need to be reshaped to be re-entrant. Hazards [VERIFIED by file read]:

| Hazard | Location | Severity |
|--------|----------|----------|
| `process.exit(1)` on error | `get-shit-done/bin/lib/core.cjs:283` (the `error()` function) | HIGH ‚Äî would tear down the worker. Must convert to thrown `Error` for in-process use. |
| `process.argv.slice(2)` | `gsd-tools.cjs:245` | HIGH ‚Äî reads global argv. Must accept argv as parameter for in-process. |
| `process.env.GSD_JSON_ERRORS / GSD_WORKSTREAM` reads | `gsd-tools.cjs:307`, `core.cjs:354,566,1217,1218` | MEDIUM ‚Äî process-shared env; in-process callers would need explicit env-snapshot |
| `fs.writeSync` patching of `process.stdout` for `--pick` and `@file:` interception | `gsd-tools.cjs:424-476` | HIGH ‚Äî monkey-patches `fs.writeSync` globally; trivially breaks concurrent in-process callers. Must lift to scoped output buffer. |
| `core.setJsonErrorMode(true)` (module-level flag `let _jsonErrorMode`) | `core.cjs:265-267` | HIGH ‚Äî process-global mutation; would affect every in-process caller. Must move into a request-scoped context. |
| `process.cwd()` default for `cwd` | `gsd-tools.cjs:248` | LOW ‚Äî overridden by `--cwd` flag or by `findProjectRoot(cwd)` |

**Cost-to-fix estimate:** Refactoring `gsd-tools.cjs` to be re-entrant is a non-trivial cross-cutting change ‚Äî touches `core.cjs` (1700+ LOC), `gsd-tools.cjs` (1208 LOC), and every command that calls `error(...)`. **An order of magnitude more code than L1+L2 combined.** Phase 03.1 is unlikely to need it. Treat L4 as a sentinel option, not a default.

**Alternative L4 path (if needed):** Substitute `captureGsdToolsOutput` ‚Üí `createRegistry().dispatch(...)` selectively in **non-golden** files. But per the audit above, those files already use the registry path. So this substitution would only affect rare slow paths (e.g., the `cliAvailable` `which claude` check at module load could be cached) and is unlikely to pay back the audit cost.

## L5 (`test.concurrent` / `describe.concurrent`) ‚Äî Likely N/A

The 7 files each contain mostly 1-3 `it(...)` per `describe`. The exception is `golden.integration.test.ts` (677 LOC, ~50 `it(...)` cases inside ~30 `describe` blocks). Within-file parallelism here is the only candidate.

### Flakiness gate (D-05c)

The 3√ó back-to-back all-pass requirement is **already covered** by the median-of-3 baseline runs (3 runs ‚Üí all must pass). The aggregator can flag "ANY run failed" as a hard error and abort commit of the baseline.

`vitest --retry <n>` exists [VERIFIED: v3.vitest.dev/guide/cli ‚Äî "Retry the test specific number of times if it fails (default: 0)"] but it is **in-process retry**, not separate-run repetition. It does not satisfy D-05c (which wants three independent runs). The shell loop is the correct primitive.

### Per-test vs per-file granularity

Per-file is required by D-08 ("median total wall-clock and per-file wall-clock"). Per-test is optional ‚Äî `assertionResults[].duration` gives it for free in the JSON. Include per-test as a secondary "slow tests" appendix in the baseline markdown if the planner has space, but it's not mandatory.

## What `pretest: pnpm run build:sdk` Actually Costs (D-07 noted but not modified)

[VERIFIED: `package.json` lines containing `"pretest": "pnpm run build:sdk"` and `"build:sdk": "pnpm --filter @gsd-build/sdk build"`]:

- Root `pnpm test` triggers `pretest`, which runs `pnpm --filter @gsd-build/sdk build` (= `tsc && pnpm run build:cjs`, two tsc passes).
- **However**, this phase's baseline command uses `pnpm --filter @gsd-build/sdk exec vitest run --project integration` (or `pnpm --filter @gsd-build/sdk run test:integration`). Neither chains through the root-level `pretest` hook.
- The `sdk/package.json` defines `"test:integration": "vitest run --project integration"` with no `pretest` of its own. So if Plan 1's baseline command is `pnpm --filter @gsd-build/sdk run test:integration`, the rebuild is **not** auto-triggered (and stale dist/ may cause confusion).
- **Recommendation for the harness:** Plan 1 should explicitly run `pnpm --filter @gsd-build/sdk run build` once before the median-of-3 loop, then loop only over `vitest run`. This isolates test wall-clock from rebuild cost and gives stable numbers.
- **One-time measurement to include in baseline notes:** `time pnpm --filter @gsd-build/sdk run build` on a warm cache. If it's <2s, ignore; if >5s, the planner knows the user-visible `pnpm test` time is dominated by something other than vitest, and ratio claims should clarify they're against vitest-only time, not full `pnpm test`.

## Common Pitfalls

### Pitfall 1: Reporter contamination across projects
**What goes wrong:** `vitest --reporter=json` at the root config level applies to **all** projects in one merged output. Per-project reporter scoping is documented as root-only.
**Why it happens:** `reporters` is one of the three documented root-only options.
**How to avoid:** Pass `--reporter=json` as a CLI flag (overrides config), AND scope the run with `--project integration`. Smoke-test once to confirm the resulting JSON's `testResults[].name` paths all match `*.integration.test.ts`.
**Warning signs:** JSON file contains `testResults` with paths matching `src/**/*.test.ts` (unit) ‚Äî means the project filter didn't apply.

### Pitfall 2: Baseline ratio confounded by rebuild cost
**What goes wrong:** Comparing `time pnpm test` pre- vs post-change attributes the rebuild slice to the wrong thing.
**Why it happens:** Root `pretest: pnpm run build:sdk` chains every `pnpm test` invocation through tsc + tsc-cjs.
**How to avoid:** Build once, then loop vitest-run-only. Document in the baseline markdown that the numbers are vitest-only.
**Warning signs:** Post-change ratio is wildly different from what the lever's prior art suggests ‚Äî likely the rebuild cost is dominating.

### Pitfall 3: Flipping `isolate: false` masks pre-existing test-order coupling
**What goes wrong:** Two integration files happen to pass independently but fail when run in worker reuse because file A imports a singleton that file B's setup mutates.
**Why it happens:** Test-order-dependent failures only surface when isolation is disabled. The bug existed all along.
**How to avoid:** After flipping L2, run the suite **with shuffled file order** at least twice. Vitest 3.x has `--shuffle` for this.
**Warning signs:** Test failures appear only on second or third median-of-3 run, or only when run alone after another file.

### Pitfall 4: Treating `--retry` as the flakiness gate
**What goes wrong:** Planner sets `--retry 3` and claims the suite is stable.
**Why it happens:** `--retry` is in-process retry. The bar (D-05c) is 3 **independent** runs.
**How to avoid:** Keep retry at 0. Use the shell loop for the gate.
**Warning signs:** `--retry` appears in any committed PR.

### Pitfall 5: Touching golden test subprocess spawn paths
**What goes wrong:** Planner refactors `captureGsdToolsOutput` to use `createRegistry()` directly to "speed it up."
**Why it happens:** Looks like a duplicated path that could be unified.
**How to avoid:** The two golden files are **the entire point** of having both paths. The spawn IS the test. Flag both files as L4-ineligible in the plan checklist.
**Warning signs:** A diff hunk shows `captureGsdToolsOutput` being replaced by `registry.dispatch` in any file under `sdk/src/golden/`.

### Pitfall 6: Per-project pool=threads but no `--no-isolate` audit on unit project
**What goes wrong:** Planner flips integration to threads, doesn't check that unit project (144 files) still has `process.chdir` calls; future maintainer copies the flag.
**Why it happens:** Per-project scoping means the unit project keeps its own pool. But that's also a footgun ‚Äî the unit project might have chdir usage that L1 on unit would break.
**How to avoid:** Per-project flips are isolated by construction. But the planner should document explicitly in vitest.config.ts that the knob lives in `projects[1]` (integration) and is intentionally absent from `projects[0]` (unit).
**Warning signs:** A future PR moves the knob to top-level `test:` block "to consolidate" ‚Äî breaks unit tests.

### Pitfall 7: `--reporter=json --outputFile` collision under parallel runs
**What goes wrong:** Two simultaneous baseline runs (e.g., from `&` in the shell loop) overwrite each other's JSON.
**Why it happens:** Single fixed `--outputFile` path.
**How to avoid:** Use `run-1.json` / `run-2.json` / `run-3.json` distinct paths in the loop. The shell loop in this RESEARCH already does so.
**Warning signs:** Aggregator script reports identical run-1 and run-3 numbers (suspicious).

```ts
// Source: pattern derived from vitest 3.x --reporter=json Jest-compatible schema
// https://v3.vitest.dev/guide/reporters
import { readFile } from 'node:fs/promises';
import { join } from 'node:path';

const dir = process.argv[2];
const runs = await Promise.all(
	[1, 2, 3].map(async (i) => JSON.parse(await readFile(join(dir, `run-${i}.json`), 'utf-8'))),
);

// Group by file path:
const byFile = new Map<string, number[]>();
for (const run of runs) {
	if (!run.success) throw new Error(`Baseline run failed ‚Äî flakiness gate blocks commit.`);
	for (const tr of run.testResults) {
		const wall = tr.endTime - tr.startTime;
		const key = tr.name.replace(/^.*\/sdk\//, 'sdk/');
		const arr = byFile.get(key) ?? [];
		arr.push(wall);
		byFile.set(key, arr);
	}
}

const median = (xs: number[]) => [...xs].sort((a, b) => a - b)[Math.floor(xs.length / 2)];

console.log('| File | Run 1 (ms) | Run 2 (ms) | Run 3 (ms) | Median (ms) |');
console.log('|------|-----------|-----------|-----------|-------------|');
let totalMedian = 0;
for (const [file, walls] of byFile) {
	const m = median(walls);
	totalMedian += m;
	console.log(`| ${file} | ${walls[0]} | ${walls[1]} | ${walls[2]} | ${m} |`);
}
console.log(`| **TOTAL** | ‚Äî | ‚Äî | ‚Äî | **${totalMedian}** |`);
```
