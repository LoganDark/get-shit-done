# Phase 2.1 Learnings: VCS Abstraction Audit — Drop Git-Only Concepts

## Summary

Phase 2.1 reshaped the public `VcsAdapter` surface to drop seven git-only concepts (pathspec, index, stage, unstage, hooks, gitDir/gitCommonDir, plus `currentBranch`→`currentBookmarks` and `expr.commit`→`expr.rev`). Nine plans landed across `phase/2.1-audit`: an atomic shape commit (plan 01) followed by eight concern-grouped caller-sweep plans (02–08) plus the close (plan 09). The phase honored Pattern 1 (shape-commit-first), held a broken-lint window for the duration, and surfaced two non-trivial backend behaviors during the CommitInput.pathspec→files migration (#3061 named-files invariant and a `status({porcelain:true})` trim regression). All 23 RESEARCH predictions held; the four open questions resolved as the planner recommended.

## Decisions

### D-21 Long-Lived Broken-Lint Branch + One-Commit-Per-Dimension
The branch ran broken-lint for 8 plans (commits `3acf0aa7` → `378241e9`). Each concern landed as exactly one source commit (D-21 atomic-per-dimension). The discipline made the dispatch table easy to audit and let the hot-path verifier read every dimension's diff in isolation.

**Rationale:** Per-concern commits keep the rebase surface clean for the eventual user-driven upstream sync and enable the same hotspot-grep audit pattern Phase 2 used (D-08 mechanical-only invariant).
**Source:** 2.1-CONTEXT.md D-21; 2.1-01..08-SUMMARY.md.

### Open Question 1 Resolution: Defer Internal `fireHook` Wiring to Phase 4
Plan 07 removed only the public `vcs.hooks` surface (`hooks.fire` / `hooks.list` / `VcsHooks` type + `hook-bridge.ts` export). Internal invocation of `.githooks/<stage>` from inside `commit()` / `push()` was NOT added in 2.1; that's Phase 4's HOOK-01..05 scope.

**Rationale:** Phase 4 owns the colocated-vs-non-colocated dispatch decision (HOOK-03); pre-emptively wiring `fireHook` in 2.1 would couple two phase decisions. Zero production callers existed, so deferral has no runtime impact.
**Source:** 2.1-RESEARCH.md Open Question 1; 2.1-07-SUMMARY.md.

### Open Question 4 Resolution: #3061 Invariant Restored via Pre-Add `read-tree HEAD`
Plan 04 deferred 3 #3061 tests when D-02 removed the `git commit -- <paths>` narrowing that previously prevented pre-staged unrelated index entries from leaking into a `commit --files` call. The fixup commit (`7f4c8ffa`) restored the invariant by inserting `git read-tree HEAD` (or `--empty` when no HEAD) before `git add -A -- <files>` in the git backend's `commit()`. Pre-staged unrelated entries now become worktree-only changes after the commit (matches jj's index-less semantics).

**Rationale:** D-02 documents the intent as jj-equivalence, not as disabling #3061 protection on git. The read-tree reset preserves the invariant without re-introducing the `-- <paths>` pathway.
**Source:** 2.1-RESEARCH.md Open Question 4; commit `7f4c8ffa`.

### Test-Only Raw-Git Synthesis Filled the Stage/Unstage Hole
After D-03 hard-removed `vcs.stage` / `vcs.unstage`, tests asserting the staged-diff response shape or synthesizing pre-staged index state had no adapter verb to use. Plan 04 established the pattern (`execGit(['add', '-A', '--', file])` after `unlink`); plan 08 reused it for git-backend.test.ts's `diff.nameStatus` probe. The pattern is lint-allowlisted via the `sdk/src/vcs/__tests__/**` and `tests/**/*.test.cjs` globs.

**Context:** When a public verb is hard-removed but a test legitimately needs to synthesize the underlying git state, raw-git in a glob-covered test file is the correct escape hatch.
**Source:** 2.1-04-SUMMARY.md, 2.1-08-SUMMARY.md.

## Patterns

### Pattern: Atomic Shape Commit Sized to Tolerable Broken-Lint Window
Plan 01's shape commit modified 6 files, induced 61 type errors, and stayed under ~10 minutes of executor time. The error count was within the bounds where downstream plans could mechanically close them concern-by-concern without losing the ability to read the diff. Sizing above ~100 induced errors would have made plan 02's mechanical sweep cognitively heavier.

**Source:** 2.1-01-SUMMARY.md; Pattern 1 from RESEARCH.

---

### Pattern: Pair Per-File Source Edits with the Same File's Tests
D-06 prescribed it; every applicable plan honored it. Plan 04 paired `sdk/src/query/commit.ts` with `commit.test.ts` in the same commit (`69406e45`); plan 06 paired `worktree-safety.cjs` with its tests; plan 08 paired source and dispatch updates with baseline-parity changes. The single-commit pairing kept the diff atomic and self-explanatory.

**Source:** 2.1-CONTEXT.md D-06; 2.1-04/06/08-SUMMARY.md.

---

### Pattern: Auto-Approve Cosmetic Re-Bless Skip When Args Unchanged
A naive re-bless run (`node tests/__tools__/capture-vcs-baselines.cjs`) regenerates every baseline, producing a diff with `captured_at` date changes and ephemeral commit SHAs on every file. When the args field is unchanged and the test uses `regex:` stdout matching, the re-bless is cosmetic noise. Plan 09 reverted the cosmetic churn and only acted on the 5 true orphans.

**Source:** 2.1-09 commit `88681c3a` decision rationale.

---

### Pattern: Auth/Gate-Free Phase = Zero Checkpoints
2.1 had zero `checkpoint:*` tasks across 9 plans. The phase ran fully autonomous from plan 01 entry to plan 09 close. No auth gates fired, no architectural decisions surfaced beyond what RESEARCH predicted (the #3061 design tension was a known carry-forward, not a Rule-4 surprise).

**Context:** Type-surface + caller sweeps are inherently checkpoint-free; every decision is deterministic from the shape commit. Phases that touch external systems (auth, infra) carry checkpoints; pure refactors don't.
**Source:** SUMMARYs 01–08 ("Auth Gates: None").

---

### Pattern: Single-Commit-Per-Plan Consolidation Per D-21
Plans 02 / 03 / 04 / 06 / 07 / 08 each landed as exactly one source commit + one summary/state doc commit. The plan's task structure (typically 2–4 tasks) was a logical decomposition for verification — not a commit boundary. D-21's "one dedicated commit per dimension" was operationalized as "consolidate task work into a single atomic source commit; ship docs in a second commit."

**Source:** 2.1-CONTEXT.md D-21; commit graph 2.1-02..2.1-08.

### Pattern: Audit-Plan-Then-No-Op
Plan 05 demonstrated that an audit-only plan (no source edits) is a valid D-21 dimension shape. The plan's purpose was to *verify* that the shape commit was complete with respect to a specific dimension (StatusEntry.index test consumers). When verification finds nothing, the SUMMARY documents the closure and the plan ships without a source commit.

**Source:** 2.1-05-SUMMARY.md.

## Surprises

### Surprise 1: `status({porcelain:true})` Trim Strips Worktree-Only-Mod Leading Space
While debugging the #3061 fixup, plan 04 (commit `7f4c8ffa`) discovered that `status({porcelain:true})` was calling `.trim()` on the porcelain stdout via `execGit`'s default behavior — which strips the LEADING space in entries like ` M file` (worktree-only modification) or ` D file` (worktree-only deletion). This corrupted the first parsed entry's path and the `.raw` byte-identity field. Fix: bypass the full `.trim()` and use `TrimEnd` only. The bug was latent (no test exercised first-entry-worktree-only) until the `#3061 amend --files` test ran with a worktree-only deletion as the first status entry, hitting `'nothing to commit'` instead of finding the entry.

**Why it surprised:** RESEARCH didn't anticipate that a generic exec-helper's trim would interact with porcelain's leading-space convention. The fix was simple (1-line TrimEnd substitution) but only surfaced because of #3061 fixup testing.
**Source:** commit `7f4c8ffa` body.

## Phase 3 Handoff (jj backend)

### Change_id Translator Placement
- `expr.rev(id)` factory (renamed from `expr.commit`) is in place with `SHA_OR_CHANGE_ID_RE` permissive regex (D-13) — accepts both git hex and jj change_id alphabet.
- `parse/git-rev.ts` and `parse/jj-rev.ts` use `rev:` brand prefix (renamed from `commit:`).
- Phase 3's jj backend translator slots into `parse/jj-rev.ts`; the regex shape and brand-prefix flow are already wired.
- D-11/D-12 context: range form (`expr.range(start, end)`) byte-equivalence to two-rev shape is established in Phase 2 (graphify.cjs is the first production consumer); jj equivalent will need revset translation but the call-site contract is fixed.

### VcsHooks Removal Implications for HOOK-01..05
- The public `vcs.hooks` surface is GONE. `VcsHooks` type and `hooks.fire` / `hooks.list` methods are deleted from `VcsAdapterCommon`.
- `hook-bridge.ts` `fireHook` is module-private (`function fireHook` not `export function`); HookStage / HookContext types are still exported for Phase 4's HOOK-03 wiring.
- HOOK-01/02/03 need to ADD internal `fireHook` calls inside `commit()` / `push()` implementations on the git backend (and the new jj backend). Currently zero production callers; the git backend's `git commit` natively fires `.git/hooks/pre-commit`. `.githooks/<stage>` firing is NOT wired in either backend post-2.1 — Phase 4 owns it.
- HOOK-04/05 (colocated-vs-non-colocated dispatch + PATH-shim deferred) inherit a clean slate; no 2.1 commitments constrain the design.

### CommitInput Contract (D-04 WC-State-Capture)
- `CommitInput` now has only `{ message, files?, amend?, noVerify? }` — `pathspec` removed.
- Git backend semantics: `git read-tree HEAD` (or `--empty` when no HEAD) → `git add -A -- <files>` → `git commit -m <msg>` (no `-a`). The `read-tree` reset preserves the #3061 named-files invariant; pre-staged unrelated entries become worktree-only changes (matches jj's index-less model).
- Phase 3's jj backend: `jj squash -B @ -k -m <msg>` is the equivalent. The `files` scope translates to jj's path argument; #3061 is structurally impossible on jj (no index).

### Status Porcelain Trim
- `status({porcelain:true})` uses `TrimEnd` (not full `.trim()`) on the porcelain stdout to preserve leading-space worktree-only-mod/del entries (` M file` / ` D file`).
- Phase 3's jj backend must match this convention — the `.raw` byte-identity field on `StatusEntry` is contract-relevant.

### Long-Lived Branch Hygiene
- `phase/2.1-audit` is mergeable to main: lint exits 0, baseline-parity 50/50 pass, full suite 1283 pass / 15 fail (every failure pre-existing on main, plan-09 confirmed via comm-diff against main full-suite run).
- The branch ran broken-lint for 8 plans; Phase 3 should consider whether a similar long-lived branch fits the jj-backend work or whether the smaller scope (one backend implementation file + the jj-rev translator) allows shorter-lived per-plan branches.
